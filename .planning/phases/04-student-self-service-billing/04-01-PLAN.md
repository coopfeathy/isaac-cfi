2. Create `app/api/student/slots/route.ts` — a public GET endpoint (no auth required per BOOK-01):
   ```typescript
   import { supabase } from '@/lib/supabase'
   import { NextResponse } from 'next/server'

   export async function GET(request: Request) {
     const { searchParams } = new URL(request.url)
     const startDate = searchParams.get('start') || new Date().toISOString()
     const endDate = searchParams.get('end') || new Date(Date.now() + 14 * 86400000).toISOString()

     const { data: slots, error } = await supabase
       .from('slots')
       .select('id, start_time, end_time, type, description, price, is_booked, instructor_id')
       .eq('is_booked', false)
       .gte('start_time', startDate)
       .lte('start_time', endDate)
       .order('start_time', { ascending: true })

     if (error) {
       return NextResponse.json({ error: error.message }, { status: 500 })
     }

     return NextResponse.json({ slots })
   }
   ```
   - Uses the public supabase client (slots table has public SELECT RLS).
   - Filters to `is_booked = false` and date range.
   - Returns `id, start_time, end_time, type, description, price, is_booked, instructor_id`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/student/slots | Public read — no auth needed |
| client -> /api/student/bookings POST | Authenticated user creates booking for themselves only |
| client -> /api/student/bookings GET | Authenticated user reads only their own bookings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | /api/student/bookings POST | mitigate | requireUser() validates Bearer token via supabase.auth.getUser(); booking user_id set from server-verified user.id, never from client body |
| T-04-02 | Tampering | /api/student/bookings POST | mitigate | slotId validated as non-empty string; slot existence and is_booked=false checked before insert; race condition mitigated by checking slot state server-side |
| T-04-03 | Repudiation | booking creation | accept | Booking record with user_id and created_at timestamp provides audit trail |
| T-04-04 | Information Disclosure | /api/student/slots GET | accept | Slot data (times, types, prices) is intentionally public for scheduling |
| T-04-05 | Denial of Service | /api/student/bookings POST | accept | No rate limiting in Phase 4 (LEAD-04/05 adds rate limiting in Phase 6); low risk for authenticated endpoint |
| T-04-06 | Elevation of Privilege | /api/student/bookings POST | mitigate | User can only book for themselves; user_id comes from auth token, not request body |
</threat_model>
